# Game Design

## Overview

把游戏设计问题拆成可验证的设计决策，优先产出具体循环、规则、变量、风险和测试方案。  
默认先理解项目现状，再给出可执行的设计，而不是只给抽象点子。

## When to Use

- 讨论核心玩法、战斗系统、职业/武器/敌人设计
- 设计或评审数值、成长、难度、奖励、反馈回路
- 输出玩法提案、战斗系统说明、平衡调优方案、原型测试计划
- 分析“为什么不好玩”“为什么某套 build 过强”“为什么 boss 不公平”
- 把模糊创意转成可做原型的规则与参数表

## First Pass

在提出方案前，先完成下面几件事：

1. 读当前仓库里已有的设计文档、原型说明、平衡表、玩法代码。
2. 明确游戏类型、目标玩家、输入方式、镜头类型、平台约束。
3. 把用户请求归类为：
   - 整体玩法/核心循环
   - 战斗系统/敌人/boss/武器/技能
   - 成长构筑/职业/属性/装备
   - 平衡与难度
   - UI 命中感/瞄准/交互精度
4. 如果上下文不完整，先做最小必要假设，并明确标注。

## Working Style

- 把“感觉”和“体验”翻译成可设计、可测试的规则。
- 优先使用具体动词、状态、资源、窗口、风险，而不是空泛术语。
- 区分目标、假设、当前值、建议值，不要混在一起。
- 对战斗系统，至少明确：
  - 玩家核心动词
  - 战斗循环
  - 资源经济
  - 压力来源
  - 反制窗口
  - 失败成本
  - 奖励与反馈
- 对平衡建议，优先给可测试区间或变量方向，例如“伤害下调 10-15%”而不是“略微降低”。
- 不把 PvP、公平竞技、单机叙事和肉鸽成长混为一谈；不同类型的平衡目标不同。

## Default Workflow

1. 定义目标体验
   - 玩家幻想是什么
   - 一局/一场战斗想让玩家经历什么情绪
   - 成功和失败分别意味着什么
2. 定义核心循环
   - 局外循环
   - 局内循环
   - 战斗瞬时循环
3. 画出战斗交换模型
   - 玩家输入
   - 系统判定
   - 敌人回应
   - 恢复与奖励
4. 明确关键变量
   - 伤害、生命、护甲、韧性、冷却、能量、距离、命中率、资源掉落
5. 检查 dominant strategy
   - 是否存在单一最优解
   - 是否存在无限控制、爆发碾压、无脑风筝、滚雪球
6. 设计测试方案
   - 哪些值先粗调
   - 哪些现象代表成功
   - 哪些信号说明系统失衡
7. 输出结果
   - 设计说明
   - 风险清单
   - 原型测试步骤
   - 下一轮迭代建议

## Output Format

默认按下面格式组织回答，除非用户明确要求别的格式：

1. 设计目标
2. 核心循环 / 战斗循环
3. 规则与变量
4. 玩家决策空间
5. 风险与失衡点
6. 调参与测试方案
7. 建议落地顺序

## Guardrails

- 不要直接给大而全的系统，先给最小可玩的原型。
- 不要只从“数值”解释问题，也要检查读图、反馈、输入容错和信息透明度。
- 不要把“更难”误当成“更有深度”。
- 动态难度只适合非强竞技场景；涉及 PvP 时优先保证可预期公平。
- 奖励系统要服务行为引导，而不是无条件堆更多掉落。
