# 一、系统目标
本系统旨在建立一套 **外观即战斗方式** 的角色构筑规则，使玩家能够通过角色外观理解其战斗方式，并通过DIY系统自由构建不同战斗玩法。
系统需要同时满足以下三点**：**
**1.视觉可读性**
玩家能够通过角色外观判断战斗方式。
**2.构筑自由度**
玩家通过装备组合生成战斗玩法，而非选择预设职业。
**3.战斗验证**
战斗体验应验证玩家对外观产生的预期。

---

# 二、核心原则
系统的核心设计原则为：
<callout emoji="bulb" background-color="light-orange" border-color="light-orange">
**外观 → 战斗预期 → 战斗验证 **
对应玩家反应如下：
玩家看到角色 → 玩家预测战斗方式 → 战斗体验验证预期
</callout>

<quote-container>
注意："外观即战斗方式"并不意味着所有外观元素都要绑定同等强度的功能，
而是强调：**更容易被识别的视觉信息，应尽量承载更高权重的战斗信息。**
</quote-container>


玩家应能够通过角色外观判断（大致把握该角色的主要战斗倾向）：
- 主要如何攻击
- 主要如何防御
- 整体战斗节奏与体验气质
如果战斗体验与视觉预期一致，则说明外观表达与玩法表达形成了有效对应。

# 三、视觉规则
<callout emoji="bulb" background-color="light-orange" border-color="light-orange">
**玩家读取角色信息存在明显的视觉优先级（视觉认知是有限资源）**
</callout>

武器、关键部件轮廓、头部等高识别度信息，更适合承载角色主要战斗方式；
装备细节、挂件、材质、染色等信息，则更多承担补充和表达作用。

<callout emoji="bulb" background-color="light-orange" border-color="light-orange">
**设计规则：系统功能权重 ∝ 视觉可读性**
系统应优先定义高识别度的关键视觉信息，并让其尽量承载更高权重的战斗信息；
其余信息识别成本更高、更加局部或装饰性，更多承担补充、修正或个性表达/功能细节作用
</callout>

# 四、系统结构
本系统采用 **构筑型结构：**
玩家通过装备结构组合，形成独特的战斗风格，而非选择预设职业或套装底子
<callout emoji="bulb" background-color="light-orange" border-color="light-orange">
**角色构筑 = 战斗结构 + 战斗参数 + 外观表达**
</callout>

<quote-container>
在初期版本中，会优先构建 **最容易被视觉识别**的性能规则，并为后续养成、技能等系统保留扩展接口
</quote-container>


同时，为保证系统可读性与可控性：
- 将通过 视觉优先级规则 保证玩家能够从外观预期玩法
- 通过 系统约束（重量 / 能量 / 资源等） 控制构筑边界

# 五、玩家构筑和装备设计（新）
## 玩家构筑
文档中对不同部件权重的区分不直接等同于玩家侧的固定构建顺序
玩家的 DIY 构建并不遵循单一、固定的选择流程，而是从**外观搭配**与**玩法体验预期**出发
在已有部件库、品质类别、个人偏好与验证需求等因素共同作用下逐步形成。

## 装备设计
**装备品质**
不同阶段和品质的装备，可能遵循不同的构造逻辑
可能有普通装备、上古装备等，品质越高获取稀有度越高

**套装和散件**
套装装备：更强调整体组合与风格表达，通常在视觉与玩法上形成较完整的构筑方向。
散件装备**：**更可能在某一类能力或玩法体验上表现得更纯粹、更突出。
在实际体验中，玩家通常不会一次性获得完整套装，而是通过逐步获得重要散件，并与已有装备进行拼接与替换，过程中逐步形成自己的构筑。

<quote-container>
这种结构能支持以下体验：
- 套装提供整体方向
- 散件提供关键能力点
- 玩家在不断获得装备的过程中逐步调整构筑
</quote-container>


**设计策略**
<callout emoji="bulb" background-color="light-orange" border-color="light-orange">
在同一品质与部件类型下，不同装备的**综合价值应大致接近**，但其能力结构可以存在明显差异
</callout>

装备之间的区别主要体现在能力侧重点，而不是简单的强弱关系。例如：
- 某些装备在防护能力上表现更强，但会明显限制机动性
- 某些装备机动性更好，但防护能力相对较弱

<callout emoji="bulb" background-color="light-orange" border-color="light-orange">
理想情况下，每件装备都应同时存在**优势场景**与**劣势场景**，使玩家根据构筑需求进行选择。
</callout>

此外装备能力差异应尽量在外观上有所体现，以帮助玩家形成战斗预期。

# 七、设计风险
**风险1：主要战斗预期的可读性**
在自由混搭前提下，角色是否仍然能够保持主要战斗预期的可读性

**风险2：灵能系统边界不清**
灵能是否改变战斗方式，比如重甲角色也可能远程攻击。 
这会导致灵能可能变成主玩法来源，稀释战斗结构的表达

**风险3：挂件系统的定位**
担忧：挂件看起来很有功能，但如果没属性会奇怪。
如果处理不好，挂件很容易变成：第二装备系统。
